# Migrations & Data Model Rules

## Alembic safety

**Always preview before applying:**
```bash
uv run alembic upgrade --sql head   # review SQL output first
uv run alembic upgrade head         # only after reviewing
```

- Never auto-apply a migration without reviewing the generated SQL
- Never squash or delete migrations that have been applied to production
- After adding a nullable→non-nullable column, always provide a server_default or a data migration step
- Migration files: `alembic/versions/` — autogenerated names are fine, but add a short description

## SQLModel / SQLAlchemy patterns

### Eager loading (N+1 prevention)
Use `selectinload` **at the call site**, not on the model relationship:

```python
select(Job)
    .options(selectinload(Job.company).selectinload(CompanyProfile.user))
    .where(Job.id == job_id)
```

Model-level `lazy="selectin"` pollutes every query that loads the parent — even list endpoints that don't need child rows. Only use model-level eager loading when the relationship is needed on **every** access path without exception.

For one-shot joins where no SQLModel relationship exists (e.g. `ActivationToken → User`), a plain `.join()` returning a tuple is fine.

### Async safety
All DB calls must be `await`ed. Never use sync SQLAlchemy sessions in async route handlers. The session is provided via `get_session` dependency — never instantiate it manually in service functions.

## Schema changes checklist
1. Write the Alembic migration (`uv run alembic revision --autogenerate -m "description"`)
2. Review the generated file — autogenerate misses some things (constraints, indexes, enum changes)
3. Run `uv run alembic upgrade --sql head` and read the output
4. Apply: `uv run alembic upgrade head`
5. Update `src/models.py` if the migration adds/removes columns
